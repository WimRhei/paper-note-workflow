# 算法论文 Markdown 模板

这份模板只规定 Algorithm 笔记的最小正文骨架。共同的 Inbox 布局、文件命名、图表命名、嵌入和清理规则以 `../SKILL.md` 的 Output Contract 为准。

只写原文支持的信息；缺失但重要的信息写 `未报告` 或 `FIX: info`。

除论文信息外，下面的章节都可以按论文逻辑改名、合并或省略。不要为了保留模板结构重复内容。

每张图只使用一句前置引导语说明它回答的问题；alt text 保留原始 `Figure N` / `Table N` 和简短标题。图后不再重复模型层次、执行链或结果。

## 模板正文

````markdown
# xxx

## 论文信息

- 标题：
- 作者与机构：
- 年份与会议/期刊/ArXiv：
- 链接：

> [!Key]
> 用一句话写论文最核心的技术判断或贡献，不复述后文结构和完整 workflow。

## 系统边界与静态模型

用最短文字界定任务、输入、输出和调用方式，再按父子关系写静态模型。训练期辅助模块、推理调度和部署组件不放入静态模型层次。

优先使用嵌套列表：

```text
系统
├── 模块 A
└── 模块 B
    ├── 子模块 B1
    └── 子模块 B2
```

只在需要逐项比较时使用表格。静态结构参数写在对应模块旁，不另建重复的参数清单。

若原文有模型或系统总览图，在此嵌入信息密度最高的一张，用它呈现模块连接和数据流；文本树不能替代该图。

## 推理执行与负载

按真实执行顺序组织，不按论文术语逐项介绍：

```text
输入
  -> 编码 / 准备
  -> 主干执行 / 循环 / 采样
  -> 解码 / 输出
```

只保留理解 workload 所需的信息：

- 数据或状态的关键尺度。
- 每轮调用的模块、次数和实际 forward。
- 串行依赖、并行或异步关系。
- cache / memory / state 的更新与复用。
- 精度、压缩、搜索、采样和部署优化对执行的影响。

相关信息应在这里形成一条执行逻辑，不再另建“推理效率信息表”重复汇总。

若核心机制依赖循环、状态变化、并发、时序或关键变换，在此嵌入对应机制图，并核对正文是否覆盖图中的关键步骤和重复调用。

## 训练机制

只保留会改变模型能力、推理结构、调用次数、state、质量或效率的训练方法。每项只在此解释一次；推理节只写它造成的运行时行为，结果节只写证据。标准推理移除的模块在此明确说明。

没有相关内容时省略本节。

## 结果与边界

- 每项结果用最少数字表达比较和结论，不复述方法或实验流程。
- 只保留影响结论解释的实验口径。
- 只列会阻碍任务、执行或 workload 判断的未报告信息。
- 推算结果显式标注“按原文数据推算”。

若原文有支撑主质量或效率结论的图表，在此嵌入信息密度最高的一张。关键数字摘要不能替代主结果的原始视觉证据。

不要默认增加“关键机制”节。机制归入它所属的模型、推理、训练或部署位置。
````
