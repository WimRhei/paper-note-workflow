# 体系结构论文 Markdown 模板

这份模板只规定 Architecture 笔记的正文结构。共同的 Inbox 布局、文件命名、图表命名、嵌入和清理规则以 `../SKILL.md` 的 Output Contract 为准，不在这里重复。

## 模板正文

```markdown
# xxx

## 论文信息

- 标题：
- 作者与机构：
- 年份与会议/期刊/ArXiv：
- 链接：

> [!Key]
> 用 2-4 句话写清楚：这篇论文解决什么问题，真实瓶颈在哪里，算法/系统分别改了什么，以及 trade-off 为什么变好。

## 背景与问题

写清楚研究对象、目标指标、原方法/系统如何运行，以及真实瓶颈在哪里。只写理解机制所需的 workflow，不要登记所有背景信息。

必要时用 3-7 步描述原 workflow：

1. ...
2. ...
3. ...

瓶颈落在第 X 步：...

如果有关键问题图，在这里就地嵌入：

![Figure N: caption 压缩版](Figure/xxx-1.png)

这张图回答的问题是：...
它支撑的判断是：...

## 方法主线

先用几句话说明这篇论文采用哪种组织逻辑。选择下面一种骨架，删掉另一种；如果都不适合，自定义更短结构。

### 选项 A：Layered Co-Design

适用于多个协同层级。层级名称必须来自论文真实机制，不预设应用/算法/硬件。

#### 关键 Observation / Insight

只保留能推出设计的观察：

- O1：...
- O2：...

#### 分层设计

Layer 1：
- 这一层的问题：
- 机制：
- 输出/约束：
- 对下一层的影响：

Layer 2：
- ...

#### 跨层协作

写清楚上层给下层什么信息/约束，下层提供什么能力/反馈。

### 选项 B：Algorithm-to-Hardware Co-Design

适用于先解决算法/方法层问题，再设计硬件/系统支撑，让收益落到端到端指标的论文。

#### 算法层

写清楚算法自身的问题、算法解法、改完得到什么可执行性质：

- 问题：
- 解法：
- 为什么成立：
- 原文给出的代价/约束：

#### 硬件 / 系统层

先讲静态组成，再讲连接关系和动态数据流。只有机制需要时才写初始化和状态更新。

```text
静态组成：模块、存储、控制器、互联
连接关系：谁给谁数据，谁控制谁
动态数据流：一次 token/tile/request/instruction 如何流过
调度关系：哪些阶段顺序、并行、流水重叠或被隐藏
存储与状态：什么数据放哪里，容量/entry/bit width 是多少
```

如果机制抽象，用一个最小例子走通：

```text
Example:
...
```

#### 算法与硬件如何接上

写清算法输出了什么性质，硬件/系统如何利用这些性质。

### 关键机制图

在对应机制处就地嵌入，不要集中堆图：

![Figure N: caption 压缩版](Figure/xxx-2.png)

这张图回答的问题是：...
它支撑的判断是：...

## 实验设置与复现实用信息

默认压缩到 5-7 行；只有复现细节本身很重要时再展开。

- 数据集 / benchmark：
- workload / trace：
- baseline：
- 硬件平台：
- 仿真器 / 编译器 / 软件栈：
- 训练或推理设置：
- 主要指标：
- 关键缺口：`FIX: info`

## 实验结果

每个实验小段形成：结论 -> 图/表 -> 读图说明 -> 关键数字 -> 这验证了技术链路中的哪一段。

### 主结果

## 原文报告的限制与适用条件

- 作者明确陈述的限制：
- 原文给出的假设和适用条件：
- 实验明确未覆盖的范围：
- 硬件/系统实现的验证范围：

不要补写原文没有报告的适用边界、未来瓶颈、优化机会或个人思考。
```
