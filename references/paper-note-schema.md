# 论文笔记 Schema

这份 schema 不是固定模板，而是论文阅读时的记录准则。目标是提醒自己：每类论文应该记录什么、按什么顺序记录、哪些信息最容易漏。

## 总体原则

- 先记录论文的本质，再补论文的叙事。
- 先讲系统或方法怎么运行，再讲作者如何包装贡献。
- 笔记不是信息登记表。作者全名单、机构全名单不要堆在开头。
- 不要写无信息量的对比或否定句；直接写真实目标、真实瓶颈和真实改法。
- 术语要准。性能、延迟、吞吐、能耗、质量、精度、成功率不要混用。
- 结构关系要准。系统链路不要写成“模块 -> 输出 -> 下游模块”，不要把输出和模块并列。
- 机制要写清执行主体（软件算法、硬件电路、编译器、调度器、轻量网络）、输入、判断条件、输出，以及为什么成立（依据、阈值、条件或物理/数学原因）。
- 量化信息能写就写，尤其是参数规模、数据规模、延迟、吞吐、面积、功耗、能耗、成功率。
- 图表不是装饰，关键图表要就地嵌入到它服务的问题、机制或实验结论旁边，并说明它回答了什么问题。
- 缺点不足要尽量写成适用边界，而不是泛泛批评。
- 没有的信息不能编造。关键机制、实验数字等必须手工校对或标记 `FIX: info`。

## 通用结构

### 论文信息

只保留必要信息，避免写成 bibtex 展开版。

- 标题
- 作者与机构：只写第一作者和通讯作者/代表性 PI，机构只写第一机构，放在一行。例：`Er Li, Yi Wang*，Tsinghua University`
- 年份与会议/期刊/ArXiv 状态，放在一行。
- 链接：只写论文正文中已经出现的链接；DOI 和 arXiv 有一个可用来源即可，正文没有就不写。

不要放在论文信息里：

- 任务领域
- 主要实验对象
- 全部作者列表
- 全部机构列表
- 多个重复来源链接

### Key

放在论文信息后面，用 `[!important]` 或 `[!Key]` 包起来。不要再单独写“核心一句话”小节。

这段只写一件事：这篇论文到底在解决什么、卡在哪里、怎么改了那个卡点。可以写 2 到 4 句，不必强行一句话。

写法：

- 用自己的话写，别复述摘要。
- 别写“提出了一个框架”这种空话。
- 别按 Introduction 的叙事顺序抄。
- 直接写成最短因果链。
- 术语只在解释机制时出现，不要让术语反过来当答案。
- 如果 `[!Key]` 已经写清楚，就不要再重复一遍。

常见压缩方式：

`对象有 A 特性；想要 B 会带来 C 代价；论文通过 D 改变资源/误差/数据/计算/控制流/硬件结构，让 B 和 C 的 trade-off 变好。`

例：

- `CREATE: 大模型有容错性，可以接受少量 bit error；降低电压能降功耗，但会引入 bit error 并伤害任务成功率。因此这篇论文本质上是在“低功耗”和“错误率/任务质量”之间做 trade-off，并用电路层清异常值、模型层压 planner outlier、应用层动态调压来应用-算法-电路的协同降压且保质。`
- `RPU: LLM 的 decode 推理瓶颈在带宽；拆分一个大 die 为多个小 die 可以增加总周长，从而挂更多 HBM 提高带宽；但现有 HBM 的带宽和容量绑定，decode 并不需要那么多容量，所以它把 HBM 的容量参数和带宽需求解耦，调容量参数从而带宽相同但容量更小。`

### 研究对象与应用场景

- 论文研究的具体对象是什么。
- 任务领域和主要实验对象在这里或后续实验设置里交代。
- 输入、输出、执行环境分别是什么。
- 这个问题为什么重要。
- 如果论文优化的是已有系统、算法或训练/推理链路，先用 3-7 步写清原 workflow，再说瓶颈位于哪个阶段。
- 目标要用准确学术词汇：性能、训练时间、推理延迟、吞吐、能耗、面积、质量、精度、成功率。
- 可补充系统假设、目标场景、workload 特点、约束来源。

### 问题本质

- 原方案卡在哪里。
- 如果前面已经写了原 workflow，这里明确瓶颈落在哪个阶段、由哪个数据流/控制流/资源约束触发，不要重新复述整条 workflow。
- 这个瓶颈属于数据、模型、算法、硬件、系统调度还是应用闭环。
- 为什么简单扩大规模不能直接解决。
- 关键名词首次出现要解释清楚。
- 最能说明问题的图嵌在这一段附近。

### 方法机制

- 方法部分聚焦论文改了哪一段、改动如何接回原链路。
- 优先按论文真实问题链组织，而不是机械分成“问题本质”“方法机制”“对应关系”三张表。
- 如果论文天然是多层/多模块方案，就按每个层级内部的“问题 -> 机制 -> 为什么成立 -> 代价/边界 -> 关键图”来写。
- workflow 只在方法本身是新 workflow 时展开；如果是原系统 workflow，放在“研究对象与应用场景”里。
- 每一步的输入输出、模块之间怎么接、哪些离线做、哪些在线做。
- 核心动作到底是在调资源、调误差、调数据、调计算还是调控制流。
- 如果是硬件实现，要写在哪个模块/电路里实现。
- 如果新增模块/模型，要写它是什么、输入输出、参数量或开销、和主模型/主硬件相比有多小。
- 如果机制依赖数学性质或 workload 性质，要写清楚依据。
- 关键图嵌在机制解释处，并说明它回答了机制中的哪个问题。

### 实验结果

- 主指标、baseline、提升幅度。
- 仿真、真机、硬件、离线数据分别是什么。
- 哪个实验证明有效，哪个证明开销，哪个证明泛化。
- 主结果图、消融图、泛化图、硬件开销表嵌在对应结论附近。
- 每个实验小段最好形成：`结论 -> 图/表 -> 读图说明 -> 数字结果 -> 这说明什么`。

### 缺点不足与适用边界

- 哪些条件下会失效。
- 是否依赖强假设。
- 是否只验证了模块，没有验证完整系统。
- 参数、阈值、结构换了以后还行不行。

### 总结思考

- 可迁移的判断方式。
- 和旧笔记里哪些工作同类、互补或冲突。
- 以后遇到类似问题先看哪里。

## 类型注意事项

### 算法 / 模型方法论文

- 任务和输入输出
- workflow
- 模型结构
- 参数规模和数据规模
- 训练方式
- 推理方式
- 核心思想
- 实验和消融
- 边界

### Arch / 系统硬件论文

- workload 或算法瓶颈
- 软件/算法层优化
- 静态硬件架构
- 存储层次和互联
- 数据流和调度
- 指令/编译/软件栈
- 面积、功耗、延迟、吞吐、能耗
- 与 GPU/ASIC/传统系统对比
- 适用边界

问题和解法要成对出现，但不一定单独写成“原平台问题与解法对应关系”小节：

- 如果论文是分层 co-design，就按层级分别写 `该层问题 -> 对应机制 -> 为什么有效 -> 结果/开销`。
- 如果论文是 pipeline 系统，就按 pipeline 阶段分别写 `该阶段瓶颈 -> 该阶段改法 -> 对下游的影响`。
- 如果论文确实是多个独立技术点，才额外用一张简短对应表兜底。

### WM / VLA / 机器人论文

- 任务闭环
- 数据来源
- 模型接口
- 训练流程
- 推理 / 执行流程
- 机器人或环境设定
- 实验任务与指标
- 失败边界

### Survey / 综述论文

- 综述范围
- 分类体系
- 领域主线
- 代表工作
- 对比标准
- 当前瓶颈
- 空白方向

## 图表记录规则

- 只嵌入关键图，不要额外在文中写 `关键图：`。
- 图表必须靠近它服务的文字：问题图放在问题本质里，架构图放在 workflow 里，机制图放在机制段里，实验图放在实验结论里。
- 每张图不只写 caption，要写它回答了什么，以及它支撑了正文里的哪一个判断。
- 优先 workflow、架构、数据流、结果、消融、失败边界。
- 不要只贴图。
- 不要在正文里写“对应图：Figure N ...”却不嵌图。
- 不要把所有图集中放到最后，除非那是附录式图表索引。
- 如果提取工具漏掉关键图，在应该嵌图的位置写 `FIX: 手工嵌入 Figure/Table N`，不要把 FIX 放到最后统一处理。

## Schema 维护规则

每次阅读新论文后，如果发现现有 schema 接不住，需要更新本文件。
