# 论文笔记 Markdown 模板

这份模板用于生成最终 Obsidian 论文笔记。它和 `paper-note-schema.md` 的关系是：

- `paper-note-schema.md` 规定阅读和判断准则。
- 本文件规定笔记落地时的章节骨架、图表写法和可复查信息位置。

生成笔记时，不要机械填空；如果某节对论文不适用，可以删掉或合并。但图表文件规则、相关工作定位、复现实用信息必须保留。

## 文件命名

- 笔记文件名使用论文提出的核心概念、系统、方法或 artifact 名称：`xxx.md`。
- `xxx` 使用简短英文或论文原名缩写，避免空格；必要时用 `-` 连接。
- 图表文件必须使用同一前缀：`xxx-1.ext`、`xxx-2.ext`、`xxx-3.ext`。
- 为兼容本地 Obsidian `paper-archiver` 插件，最终落盘必须是 `03-论文阅读/Inbox/xxx/xxx.md`。
- 图表文件放在 `03-论文阅读/Inbox/xxx/Figure/`，源 PDF 直接放在 `03-论文阅读/Inbox/xxx/`，通常命名为 `xxx.pdf`。

## 图表文件规则

图表文件规则是强约束，不要按个人习惯改写。

### 命名

- 所有嵌入笔记的图表文件都放在笔记相邻的 `Figure/` 目录中。
- 文件名格式必须是：`xxx-N.ext`。
- `xxx` 必须和笔记文件名完全一致。
- `N` 从 1 开始，按图表在笔记中首次出现的顺序递增。
- `ext` 保留实际文件格式，例如 `png`、`jpg`、`webp`。
- 不允许使用原始文件名，例如 `Figure_1.png`、`table-2.png`、`image.png`。
- 不允许跳号、重号或按论文原 Figure/Table 编号命名。

### 嵌入格式

每张图表必须就地嵌入到它支撑的段落附近。推荐格式：

```markdown
![Figure 2: 原论文 caption 的压缩版](Figure/xxx-1.png)

这张图回答的问题是：...
它支撑的判断是：...
```

不要使用 `attachments/`。`paper-archiver` 归档后会把图片移动到主题目录的 `Figure/` 下，因此正文必须使用 `Figure/xxx-N.ext` 这样的相对路径。

### 原编号保留

- Markdown alt 文本必须保留原论文编号，例如 `Figure 1`、`Fig. 3`、`Table 2`。
- 文件名中的编号只表示笔记出现顺序，不表示原论文编号。
- 正文说明中要写清该图/表回答的问题，不能只复述 caption。

### 缺失图表

如果 `pdffigures2` 没提取出关键图表，必须在应嵌入的位置写：

```markdown
FIX: 手工嵌入 Figure/Table N
```

不要把所有缺失图表统一放到文末处理。

## 模板正文

```markdown

## 论文信息

- 标题：
- 作者与机构：
- 年份与会议/期刊/ArXiv：
- 链接：

> 用 2-4 句话写清楚：这篇论文解决什么问题，真实瓶颈在哪里，它改了哪一个关键环节，以及 trade-off 为什么变好。

- 它延续的是哪条研究线：
- 它反对或补足的旧方案是什么：
- 和最接近工作的关键差异：
- 这篇论文真正新增的判断、机制或工程约束：

## 问题是什么

这一节只回答论文要解决的核心问题，不展开解法。

- 目标对象：
- 目标状态：
- 原方案为什么达不到：
- 如果不解决，会造成什么具体后果：

如果有关键问题图，在这里就地嵌入：

### C1：

- 挑战是什么：

| --- | --- | --- | --- |
| C1 | M1 | ... | ... |

### M1：

- 对应挑战：C1
- 改了原链路的哪一段：
- 执行主体：
- 输入：
- 判断条件或触发条件：
- 输出：
- 为什么成立：
- 代价或边界：

必要时嵌入机制图：

它支撑的判断是：...

### M2：

- 对应挑战：C2

- 代价或边界：

这节集中记录复现、复查和横向比较需要的信息。

- 代码是否公开：
- 数据集 / benchmark：
- workload / trace：
- baseline：
- 硬件平台：
- 仿真器 / 编译器 / 软件栈：
- 模型规模 / 参数规模：
- 训练或推理设置：
- 关键超参 / 阈值：
- 主要指标：
- 复现门槛：
- 论文没有交代但复现需要的信息：`FIX: info`

每个实验小段尽量形成：结论 -> 图/表 -> 读图说明 -> 数字结果 -> 这说明什么。结果统一放在本节，不要散落到问题或方法章节；但每个结果要说明它验证了哪个挑战或机制，例如 `验证：C1/M1`。
```
